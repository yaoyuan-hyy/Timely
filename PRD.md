# Timely 产品需求文档 (PRD)

> 一个用自然语言记录时间点事件与账目的个人记录 App
> 版本：v0.4 | 2026-06-10

---

## 1. 产品定位

Timely 的目标只有一个：**记录**。

用户用自然语言告诉 Timely 要记录什么，Timely 解析出时间点和内容，然后保存到本地记录库。记录可以发生在未来，也可以发生在过去。

当前阶段先做好 **事件记录**。账目记录是同一产品方向下的后续独立记录类型，本阶段不实现。

### 一句话定位

Timely 是一个面向个人的自然语言记录 App，帮助用户快速记录过去或未来某个时间点发生的事情。

### 当前阶段范围

本阶段只做事件记录闭环：

```text
用户输入一句自然语言
  -> 本地解析器提取事件标题和时间点
  -> 程序写入 CalendarEvent
  -> 用户在事件记录列表中查看
```

示例：

```text
用户：帮我记录一下6月13号下午3点有一个会议
Timely：已记录。6月13日 15:00，会议。
```

```text
用户：帮我记录6月9日下午4点复盘
Timely：已记录。6月9日 16:00，复盘。
```

### 产品原则

- **记录优先**：只保存用户明确表达的事情，不替用户规划。
- **过去和未来都能记**：过去时间点不是错误，不需要二次确认。
- **用户说什么就记什么**：不擅自扩展为任务、提醒、建议或总结。
- **只在必要时反问**：缺少事件标题或明确时间点时才反问。
- **本地优先**：MVP 数据保存在浏览器本地。
- **确认简洁**：成功后用一句话确认关键结果。

## 2. MVP 范围

### P0：事件记录必须实现

#### 2.1 对话录入

- 支持文字输入。
- 支持用户用自然语言创建事件记录。
- 支持一次输入识别一个主要事件。
- 支持一次任务内的短澄清上下文，例如：

```text
用户：帮我记录一个会议
Timely：什么时候？
用户：6月13日下午3点
Timely：已记录。6月13日 15:00，会议。
```

#### 2.2 本地解析

- 识别事件标题。
- 识别明确时间点。
- 支持绝对日期，例如 `2026年6月13日`。
- 支持无年份日期，例如 `6月13日`，默认使用当前年份。
- 支持相对日期，例如 `今天`、`明天`、`昨天`。
- 支持常见时间表达，例如 `下午3点`、`15:00`、`晚上8点半`。
- 缺少时间点时反问 `什么时候？`。
- 缺少标题时反问 `记录什么？`。

#### 2.3 事件保存与展示

- 新增 `CalendarEvent`。
- 刷新页面后记录不丢失。
- 事件记录页按日期和时间展示 active 事件。
- 用户可以在事件记录页取消单条记录。
- 取消记录使用状态更新，不直接硬删除核心记录。

#### 2.4 回复风格

- 成功记录：`已记录。6月13日 15:00，会议。`
- 缺少时间：`什么时候？`
- 缺少标题：`记录什么？`
- 未识别：`我可以帮你记录事件。`

### P1：后续再做

- 查询某天记录。
- 修改事件记录。
- 更完整的中文时间解析。
- 搜索和标签。
- 账目记录：作为独立记录类型设计，不混入事件记录模型。
- 数据导出。
- 多端同步。

### 明确不做

- 当前阶段不做提醒。
- 当前阶段不做时间规划、优先级排序、番茄钟或效率洞察。
- 当前阶段不做系统日历同步。
- 当前阶段不做长期上下文引用，例如“刚才那个改一下”。
- 当前阶段不做 AI 主动建议、主动总结或主动推送。

## 3. 数据模型

### 3.1 CalendarEvent

```ts
type CalendarEvent = {
  id: string;
  title: string;
  startsAt: string;
  endsAt: string | null;
  location: string | null;
  notes: string | null;
  status: "active" | "cancelled";
  sourceText: string;
  createdAt: string;
  updatedAt: string;
};
```

### 3.2 ConversationMessage

```ts
type ConversationMessage = {
  id: string;
  role: "user" | "assistant";
  content: string;
  createdAt: string;
};
```

### 3.3 PendingClarification

```ts
type PendingClarification = {
  kind: "event_time";
  title: string;
  sourceText: string;
  createdAt: string;
};
```

### 3.4 TimelyState

```ts
type TimelyState = {
  events: CalendarEvent[];
  reminders: Reminder[];
  messages: ConversationMessage[];
  pendingClarification: PendingClarification | null;
};
```

`reminders` 暂时保留为空数组，避免后续迁移成本；当前 UI 和业务流不使用提醒。

## 4. 时间规则

- 默认时区为 `Asia/Shanghai`。
- 所有相对时间以用户设备当前时间为基准。
- 无年份日期使用当前年份。
- 无年份日期即使已经过去，也直接记录为当前年份的过去日期。
- `上午`、`早上` 不改写小时。
- `下午`、`晚上`、`傍晚` 遇到 1-11 点时转换为 13-23 点。
- `中午1点` 解析为 13:00。
- 只有日期、没有具体时间时，需要反问 `什么时候？`。

## 5. MVP 验收用例

| 编号 | 用户输入 | 预期结果 |
| --- | --- | --- |
| 1 | 帮我记录一下6月13号下午3点有一个会议 | 创建事件：会议，2026-06-13 15:00 |
| 2 | 6月13日下午3点有个会议 | 创建事件：会议，2026-06-13 15:00 |
| 3 | 帮我记录6月9日下午4点复盘 | 创建过去事件：复盘，2026-06-09 16:00 |
| 4 | 昨天晚上8点看电影 | 创建过去事件：看电影，昨天 20:00 |
| 5 | 帮我记录一个会议 | 不创建记录，反问 `什么时候？` |
| 6 | 用户补充：6月13日下午3点 | 使用待澄清标题创建事件 |
| 7 | 6月13日下午3点 | 不创建记录，反问 `记录什么？` |
| 8 | 最近怎么样 | 不闲聊，回复 `我可以帮你记录事件。` |

## 6. 成功定义

当前阶段完成时，应满足：

- 用户能在 10 秒内通过一句话创建事件记录。
- 过去和未来日期都能保存。
- 缺少时间时不会写入脏数据。
- 澄清一次后能补全并保存事件。
- 事件记录页能看到新增记录。
- 刷新页面后记录仍存在。
- 界面和文档不再把 Timely 表达为提醒、规划或效率工具。
