# Timely 产品需求文档 (PRD)

> 一个用自然语言完成提醒与轻量记账的个人助手
> 版本：v0.2 | 2026-06-07

---

## 1. 产品概述

### 一句话定位
Timely 是一个用文字对话完成提醒与轻量记账的个人工具。第一版不依赖长期上下文，只解析当前输入与必要的澄清对话；它沉默克制，除到点提醒外不主动打扰用户。

### 产品哲学
- **用户说什么就记什么**：不擅自补充用户没有表达的信息。
- **沉默是金**：除用户创建的提醒外，不主动推送任何信息。
- **只在必要时反问**：信息缺失到无法准确执行时，才反问澄清。
- **不做建议者，只做执行者**：不主动提供消费建议、日程建议或内容推荐。
- **确认感优先**：每次成功执行后，用一句话确认关键结果。

### MVP 平台范围
- 第一版优先做 **Web 端 MVP**。
- 通知能力优先使用浏览器 Notification API。
- iOS 原生 App、iOS 原生通知、Apple Watch、系统日历集成均不属于 MVP。

### 目标用户
- 25-35 岁都市白领 / 在读研究生。
- 同时使用多个 App 管理提醒、备忘和记账。
- 讨厌手动输入，愿意用一句自然语言完成记录。
- 注重隐私和个人数据掌控。

### 核心价值
- 一句话完成创建提醒、记账，或两者组合。
- 不用打开多个 App，不用手动选择时间和分类。
- AI 只在必要时介入，大多数时候安静记录和确认。

## 2. MVP 范围

### P0：必须实现

#### 2.1 对话录入
- 支持文字输入。
- 支持一次输入识别一个或多个意图。
- 支持一次任务内的短澄清上下文，例如用户先说“提醒我体检”，Timely 反问“什么时候？”，用户回答“下周三上午”。
- 不支持跨任务上下文引用，例如“刚才那个改成周四”。

#### 2.2 AI 解析
- 识别意图：创建提醒、修改提醒、删除提醒、查询提醒、创建账单、查询账单、混合意图、未知意图。
- 识别时间：绝对时间、相对时间、自然语言时间。
- 提取提醒标题、地点、金额、分类、备注。
- AI 只负责解析，不直接操作数据库。
- AI 必须输出结构化 JSON，由程序根据 JSON 执行。

#### 2.3 提醒管理
- 新增提醒。
- 查询提醒。
- 删除提醒。
- 修改提醒的时间和标题。
- 到点触发浏览器通知；若通知权限被拒绝，提醒仍保存，并在 App 内展示待提醒状态。

#### 2.4 基础记账
- 一句话记录实际支出。
- 自动分类。
- 展示账单明细列表。
- 展示本月实际支出总额和分类汇总。
- 区分“实际支出”和“预算/预计支出”。

#### 2.5 沉默原则
- 不主动推送天气、新闻、内容推荐、消费建议。
- 不主动做月度总结，除非用户主动查询。
- 不主动发起对话。
- 成功回复尽量一句话完成。

### P1：重要但不急
- 语音输入与自动转写。
- 重复提醒，例如“每周三晚上健身”。
- 账单修改与删除。
- 月度账单图表。
- 标签和搜索。
- 数据导出。
- 多端同步。

### P2：未来考虑
- 接入 iOS 原生通知。
- 接系统日历，例如 iOS EventKit / Google Calendar。
- 账单导入，例如微信 / 支付宝账单。
- Apple Watch / 桌面小组件。
- 用户主动询问时的消费洞察。
- 多人协作，例如家庭账本。

### 明确不做
- MVP 不做语音输入。
- MVP 不做长期上下文引用，例如“刚才那个”。
- MVP 不接系统日历。
- MVP 不做主动建议、主动推荐、主动总结。
- MVP 不做多人协作 / 社交功能。
- MVP 不做付费会员 / 增值服务。

## 3. 用户场景

### 场景 1：完整提醒输入
```
用户：明天下午3点提醒我取快递
Timely：已提醒。明天下午3点，取快递。
```

### 场景 2：提醒信息不完整
```
用户：提醒我体检
Timely：什么时候提醒？
用户：下周三上午
Timely：已提醒。下周三上午9点，体检。
```

### 场景 3：纯记账
```
用户：刚打车花了25
Timely：已记账。交通，25元。
```

### 场景 4：预算记录
```
用户：下周日跟朋友吃饭，预计人均150
Timely：已记录预算。下周日，餐饮，150元。
```

### 场景 5：提醒 + 预算
```
用户：下周日跟朋友吃饭，预计人均150，提前一天提醒我
Timely：已提醒。下周六，朋友吃饭；已记录预算：餐饮，150元。
```

### 场景 6：修改提醒
```
用户：明天下午3点的会议改到4点
Timely：已改为明天下午4点，会议。
```

### 场景 7：删除提醒，多个匹配时反问
```
用户：取消明天的会议
Timely：找到2个明天的会议：
1. 15:00 内部会议
2. 17:00 客户会议
取消哪个？
```

### 场景 8：删除提醒，单个匹配时执行
```
用户：取消明天3点的会议
Timely：已取消。明天15:00，内部会议。
```

### 场景 9：查询提醒
```
用户：明天下午有什么安排
Timely：明天下午：
1. 15:00 内部会议
2. 17:00 客户会议
```

### 场景 10：未知意图
```
用户：最近怎么样
Timely：我可以帮你记录提醒或账单。
```

## 4. AI 解析与执行规则

### 4.1 总原则
- AI 只解析用户输入，不能直接写数据库。
- 程序只根据结构化 JSON 执行。
- `needs_clarification` 为 `true` 时，不写入数据库。
- `intent` 为 `unknown` 时，不执行任何写入、修改或删除。
- 如果解析结果存在高风险歧义，例如删除或修改匹配多条记录，必须反问。

### 4.2 AI 输出格式
```json
{
  "intent": "create_reminder | update_reminder | delete_reminder | query_reminder | create_expense | query_expense | mixed | unknown",
  "needs_clarification": false,
  "clarification_question": null,
  "reply": "已提醒。明天下午3点，取快递。",
  "reminder": {
    "title": "取快递",
    "datetime": "2026-06-08T15:00:00+08:00",
    "location": null,
    "match_hint": null,
    "raw_text": "明天下午3点提醒我取快递"
  },
  "expense": {
    "type": "actual",
    "amount": 25,
    "currency": "CNY",
    "category": "交通",
    "note": "打车",
    "datetime": "2026-06-07T12:00:00+08:00",
    "raw_text": "刚打车花了25"
  }
}
```

### 4.3 字段说明
- `intent`：用户的主意图；混合意图使用 `mixed`。
- `needs_clarification`：是否需要反问。
- `clarification_question`：只允许出现一个问题。
- `reply`：给用户看的简洁回复。
- `reminder.datetime`：必须是 ISO 8601 字符串。
- `expense.type`：`actual` 表示实际支出，`budget` 表示预算或预计支出。
- 缺失字段使用 `null`，不要编造。

### 4.4 反问规则
- 创建提醒缺少时间时，必须反问时间。
- 创建提醒缺少标题时，可以使用用户原话作为标题，不反问。
- 记账缺少金额时，必须反问金额。
- 记账缺少分类时，默认归入“其他”，不反问。
- 删除或修改提醒匹配多条时，必须列出选项并反问。
- 一次只反问一个问题。

### 4.5 回复风格
- 成功执行：一句确认 + 关键信息。
- 信息缺失：只问一个问题。
- 执行失败：说明原因，不扩展建议。
- 不使用营销话术，不主动展开闲聊。

示例：
```
已提醒。明天下午3点，取快递。
已记账。交通，25元。
什么时候提醒？
没找到明天下午3点的会议。
```

## 5. 时间、金额与分类规则

### 5.1 时间规则
- 所有相对时间以用户设备当前时间为基准。
- 默认时区为 `Asia/Shanghai`。
- “上午”默认 09:00。
- “下午”默认 15:00。
- “晚上”默认 20:00。
- “晚饭前”默认 18:00。
- 只有日期、没有具体时间或时段时，需要反问具体时间。
- 解析出的提醒时间如果已经过去，需要反问用户确认新时间，不得直接创建过期提醒。
- “提前一天提醒我”表示在事件日期前一天 09:00 创建提醒；如果用户给出具体提前时间，以用户表达为准。

### 5.2 金额规则
- 默认货币为人民币，`currency` 使用 `CNY`。
- “花了/付了/买了/打车25”识别为实际支出 `actual`。
- “预计/预算/大概/人均”识别为预算 `budget`。
- 预算不计入本月实际支出总额，但可以在账单页单独展示。
- 金额缺失时，不创建账单，反问“花了多少？”。
- 金额可以识别中文数字和阿拉伯数字，例如“二十五”“25”“25元”。

### 5.3 分类规则
- 能明确分类时自动分类。
- 无法明确分类时归入“其他”。
- MVP 不支持自定义分类。

分类列表：
- **餐饮**：外卖、堂食、咖啡、奶茶、聚餐。
- **交通**：打车、公交、地铁、加油、停车。
- **购物**：日用品、衣服、数码、化妆品。
- **住房**：房租、水电、物业。
- **医疗**：药品、挂号、体检。
- **教育**：书籍、课程、培训。
- **娱乐**：电影、游戏、聚会。
- **通讯**：话费、流量、网费。
- **其他**：以上未覆盖的支出。

## 6. 数据模型

### 6.1 Reminder
```ts
type Reminder = {
  id: string;
  title: string;
  remindAt: string;
  location: string | null;
  status: "active" | "done" | "cancelled";
  sourceText: string;
  createdAt: string;
  updatedAt: string;
  notifiedAt: string | null;
};
```

### 6.2 Expense
```ts
type Expense = {
  id: string;
  amount: number;
  currency: "CNY";
  category: "餐饮" | "交通" | "购物" | "住房" | "医疗" | "教育" | "娱乐" | "通讯" | "其他";
  type: "actual" | "budget";
  note: string;
  spentAt: string;
  sourceText: string;
  createdAt: string;
  updatedAt: string;
};
```

### 6.3 ConversationMessage
```ts
type ConversationMessage = {
  id: string;
  role: "user" | "assistant";
  content: string;
  createdAt: string;
};
```

### 6.4 本地存储策略
- MVP 可使用本地数据库或浏览器本地存储实现。
- 用户提醒、账单、对话消息保存在本地。
- 若接入云端 AI 服务，只发送当前用户输入、当前时间、时区和必要的澄清上下文。
- 不上传完整提醒库和完整账单库。

## 7. 业务流程

### 7.1 新建提醒
```
用户输入
  ↓
AI 解析为结构化 JSON
  ↓
检查是否缺少提醒时间
  ↓
完整 → 写入 Reminder → 返回确认
缺失 → 反问时间 → 用户补充 → 重新解析
```

### 7.2 修改提醒
```
用户输入
  ↓
AI 解析修改意图和匹配条件
  ↓
查询 Reminder
  ↓
0 个匹配 → 返回未找到
1 个匹配 → 更新 Reminder → 返回确认
多个匹配 → 列出选项 → 用户选择 → 更新
```

### 7.3 删除提醒
```
用户输入
  ↓
AI 解析删除意图和匹配条件
  ↓
查询 Reminder
  ↓
0 个匹配 → 返回未找到
1 个匹配 → 标记为 cancelled → 返回确认
多个匹配 → 列出选项 → 用户选择 → 标记为 cancelled
```

### 7.4 查询提醒
```
用户输入
  ↓
AI 解析时间范围
  ↓
查询 active Reminder
  ↓
按时间升序展示
```

### 7.5 创建账单
```
用户输入
  ↓
AI 解析金额、分类、类型
  ↓
金额完整 → 写入 Expense → 返回确认
金额缺失 → 反问金额
```

### 7.6 提醒触达
```
定时检查 active Reminder
  ↓
找到 remindAt <= 当前时间 且 notifiedAt 为空的提醒
  ↓
触发浏览器通知
  ↓
写入 notifiedAt
```

## 9. 非功能需求

### 9.1 性能
- 对话响应时间：P95 < 3 秒。
- 本地数据库查询：P95 < 100ms。
- 通知检查延迟：P95 < 60 秒。

### 9.2 安全与隐私
- 本地保存提醒、账单和对话消息。
- API 调用使用 HTTPS。
- 调用 AI 服务时，只发送当前输入、当前时间、时区和必要澄清上下文。
- 不上传完整账单库、完整提醒库。
- 设置页提供清除本地数据入口。

### 9.3 可用性
- 用户可以在 10 秒内完成一条提醒创建。
- 用户可以在 10 秒内完成一条账单记录。
- 反问只问一个问题。
- 回复简洁，不主动延伸话题。

### 9.4 兼容性
- 支持现代桌面和移动浏览器。
- 支持深色模式。
- Web Notification 不可用时，功能降级为 App 内提醒状态展示。

## 12. 风险与挑战

### 12.1 技术风险
- **AI 解析不稳定**：中文时间和金额表达多样，可能识别错。
  - 缓解：使用结构化 JSON、明确时间规则、补充验收用例。
- **通知能力受浏览器限制**：不同浏览器对通知权限和后台触达支持不同。
  - 缓解：MVP 提供 App 内提醒状态作为降级方案。
- **隐私与 AI API 冲突**：用户输入需要发给 AI 服务解析。
  - 缓解：只发送当前输入和必要上下文，不上传完整数据库。

### 12.2 产品风险
- **提醒和记账组合感不强**：用户可能觉得是两个工具拼在一起。
  - 缓解：强调“用户不用判断打开哪个 App，只要说一句话”。
- **沉默原则过度**：用户可能觉得 AI 太冷。
  - 缓解：成功确认要明确，反问语气友好，但不主动扩展。

### 12.3 求职展示风险
- **功能做得多但不深**：容易变成浅层 Demo。
  - 缓解：P0 聚焦文字输入、结构化解析、提醒、记账、本地存储和可演示闭环。

## 13. 成功指标

### 13.1 求职场景
- 能演示 5 个核心流程：创建提醒、缺失时间反问、创建账单、预算记录、删除/修改提醒。
- 至少 3 个内测用户反馈“这个能用”。
- 能在面试中清晰讲出：产品定位、MVP 边界、AI 解析方案、数据模型、隐私策略、迭代计划。

### 13.2 产品场景
- 内测用户次日留存 > 50%。
- 平均每个用户每天记录 3 次以上。
- 30% 以上内测用户愿意推荐给朋友。

## 14. 附录

### 14.1 术语表
- **PRD**：Product Requirements Document，产品需求文档。
- **MVP**：Minimum Viable Product，最小可行产品。
- **反问**：AI 在信息不完整时主动询问用户。
- **沉默原则**：不主动推送任何用户没问的信息。
- **实际支出**：已经发生并计入月度统计的消费。
- **预算**：预计发生但尚未实际支出的金额，不计入实际支出统计。
