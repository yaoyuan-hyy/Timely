# Timely 产品需求文档 (PRD)

> 一个用自然语言完成日程记录与提醒的个人助手
> 版本：v0.3 | 2026-06-09

---

## 1. 产品概述

### 一句话定位
Timely 是一个提醒和记录 App。用户用自然语言告诉 App 要记录什么，内置 AI Agent 解析后，把事件保存到 App 内部日历，并在需要时创建提醒。

### 当前定位校准
Timely **不是**帮助用户规划时间、安排优先级、做番茄钟或生成效率建议的 App。

Timely 的核心动作是：
- 用户说一句话。
- AI Agent 理解要记录的事件、时间、地点和提醒需求。
- App 把结果写入内部日历或提醒列表。
- App 用简洁回复确认记录结果。

典型输入：
```text
帮我记录一下6月13号下午3点有一个会议
```

典型结果：
```text
已记录。6月13日下午3点，会议。
```

### 产品哲学
- **用户说什么就记什么**：不替用户规划，不擅自补充用户没有表达的信息。
- **记录优先**：核心是保存用户明确表达的事件、提醒和备忘。
- **沉默克制**：除用户设置的提醒外，不主动推送任何信息。
- **只在必要时反问**：缺少关键时间、事件标题或出现高风险歧义时，才反问澄清。
- **确认感优先**：每次成功执行后，用一句话确认关键结果。

### MVP 平台范围
- 第一版优先做 **手机界面优先的 Web/PWA 客户端 MVP**。
- 数据保存在 App 内部，本地存储优先。
- 通知能力优先使用浏览器 Notification API。
- iOS 原生 App、系统日历集成、Apple Watch、多端同步均不属于 MVP。

### 目标用户
- 经常需要快速记录会议、待办、提醒、备忘的人。
- 讨厌手动打开日历、选择日期、选择时间、填写标题。
- 希望一句话就能把事情保存下来。
- 注重隐私和个人数据掌控。

### 核心价值
- 一句话完成日程记录和提醒创建。
- 不需要用户手动填写表单。
- AI 只负责理解和转结构化数据，最终执行由程序完成。
- App 内部日历成为用户记录和查看未来事项的中心。

## 2. MVP 范围

### P0：必须实现

#### 2.1 对话录入
- 支持文字输入。
- 支持用户用自然语言创建日历事件，例如“帮我记录一下6月13号下午3点有一个会议”。
- 支持用户用自然语言创建提醒，例如“明天下午3点提醒我取快递”。
- 支持一次输入识别一个主要意图。
- 支持一次任务内的短澄清上下文，例如用户先说“帮我记录一个会议”，Timely 反问“什么时候？”，用户回答“6月13日下午3点”。
- 不支持跨任务长期上下文引用，例如“刚才那个改成周四”。

#### 2.2 AI Agent 解析
- 识别意图：创建日历事件、创建提醒、修改记录、删除记录、查询日历、未知意图。
- 识别时间：绝对时间、相对时间、自然语言时间。
- 提取标题、开始时间、结束时间、地点、备注、是否需要提醒。
- AI Agent 只负责解析，不直接操作数据库或本地存储。
- AI Agent 必须输出结构化 JSON，由程序根据 JSON 执行。

#### 2.3 内部日历记录
- 新增日历事件。
- 查询日历事件。
- 修改事件的时间、标题、地点和备注。
- 删除或取消事件。
- 按日期和时间展示 App 内部日历。
- 事件默认保存在 App 内部，不同步到系统日历。

#### 2.4 提醒管理
- 支持创建独立提醒。
- 支持为日历事件设置提醒时间。
- 到点触发浏览器通知。
- 如果通知权限被拒绝，记录仍然保存，并在 App 内展示待提醒状态。
- 支持完成、取消和删除提醒。

#### 2.5 查询与确认
- 支持查询某天、某个时间段、未来事项。
- 查询结果按时间升序展示。
- 成功回复尽量一句话完成。
- 不主动提供日程优化建议、优先级建议或时间规划建议。

### P1：重要但不急
- 语音输入与自动转写。
- 重复事件或重复提醒，例如“每周三下午开会”。
- 标签和搜索。
- 数据导出。
- 多端同步。
- 接入系统日历，例如 iOS Calendar / Google Calendar。
- 轻量记账。如果后续继续保留该方向，应作为独立模块，不影响 P0 的日历记录闭环。

### P2：未来考虑
- iOS 原生通知。
- Apple Watch / 桌面小组件。
- 更复杂的日程视图，例如周视图、月视图。
- 用户主动询问时的统计回顾。
- 多人协作，例如共享家庭日历。

### 明确不做
- MVP 不做时间规划、任务优先级排序、番茄钟、效率洞察。
- MVP 不主动建议用户如何安排时间。
- MVP 不接系统日历。
- MVP 不做长期上下文引用，例如“刚才那个”。
- MVP 不主动发起对话、主动总结或主动推荐。
- MVP 不做多人协作 / 社交功能。
- MVP 不做付费会员 / 增值服务。

## 3. 用户场景

### 场景 1：记录日历事件
```text
用户：帮我记录一下6月13号下午3点有一个会议
Timely：已记录。6月13日下午3点，会议。
```

### 场景 2：记录信息不完整
```text
用户：帮我记录一个会议
Timely：什么时候？
用户：6月13日下午3点
Timely：已记录。6月13日下午3点，会议。
```

### 场景 3：创建提醒
```text
用户：明天下午3点提醒我取快递
Timely：已提醒。明天下午3点，取快递。
```

### 场景 4：为事件设置提前提醒
```text
用户：6月13日下午3点有个会议，提前半小时提醒我
Timely：已记录。6月13日下午3点，会议；提前半小时提醒。
```

### 场景 5：查询日历
```text
用户：6月13号有什么安排
Timely：6月13日：
1. 15:00 会议
```

### 场景 6：修改记录
```text
用户：6月13日下午3点的会议改到4点
Timely：已改为6月13日下午4点，会议。
```

### 场景 7：删除记录，多个匹配时反问
```text
用户：取消6月13号的会议
Timely：找到2个6月13日的会议：
1. 15:00 内部会议
2. 17:00 客户会议
取消哪个？
```

### 场景 8：未知意图
```text
用户：最近怎么样
Timely：我可以帮你记录日程或提醒。
```

## 4. AI Agent 解析与执行规则

### 4.1 总原则
- AI Agent 只解析用户输入，不能直接写数据库。
- 程序只根据结构化 JSON 执行。
- `needs_clarification` 为 `true` 时，不写入数据。
- `intent` 为 `unknown` 时，不执行任何写入、修改或删除。
- 删除或修改匹配多条记录时，必须反问。
- AI 不能把用户输入扩展成规划建议。

### 4.2 AI 输出格式
```json
{
  "intent": "create_event | create_reminder | update_event | delete_event | query_calendar | unknown",
  "needs_clarification": false,
  "clarification_question": null,
  "reply": "已记录。6月13日下午3点，会议。",
  "event": {
    "title": "会议",
    "starts_at": "2026-06-13T15:00:00+08:00",
    "ends_at": null,
    "location": null,
    "notes": null,
    "reminder_offset_minutes": null,
    "match_hint": null,
    "raw_text": "帮我记录一下6月13号下午3点有一个会议"
  },
  "reminder": {
    "title": "取快递",
    "remind_at": "2026-06-10T15:00:00+08:00",
    "related_event_id": null,
    "raw_text": "明天下午3点提醒我取快递"
  },
  "query": {
    "range_start": "2026-06-13T00:00:00+08:00",
    "range_end": "2026-06-14T00:00:00+08:00"
  }
}
```

### 4.3 字段说明
- `intent`：用户主意图。
- `needs_clarification`：是否需要反问。
- `clarification_question`：只允许出现一个问题。
- `reply`：给用户看的简洁回复。
- `event.starts_at`：事件开始时间，必须是 ISO 8601 字符串。
- `event.ends_at`：事件结束时间。用户未表达时使用 `null`，不要编造。
- `event.reminder_offset_minutes`：相对事件开始时间的提前提醒分钟数，例如提前半小时为 `30`。
- `reminder.remind_at`：独立提醒的触达时间，必须是 ISO 8601 字符串。
- `query.range_start` / `query.range_end`：查询时间范围。
- 缺失字段使用 `null`。

### 4.4 反问规则
- 创建日历事件缺少时间时，必须反问时间。
- 创建日历事件缺少标题时，可以使用用户原话提炼短标题；无法提炼时反问“记录什么？”。
- 创建提醒缺少提醒时间时，必须反问时间。
- 修改或删除记录匹配多条时，必须列出选项并反问。
- 解析出的时间如果已经过去，需要反问用户确认新时间，不得直接创建过期记录或过期提醒。
- 一次只反问一个问题。

### 4.5 回复风格
- 成功执行：一句确认 + 关键信息。
- 信息缺失：只问一个问题。
- 执行失败：说明原因，不扩展建议。
- 不使用营销话术，不主动闲聊。

示例：
```text
已记录。6月13日下午3点，会议。
已提醒。明天下午3点，取快递。
什么时候？
没找到6月13日下午3点的会议。
```

## 5. 时间规则

- 所有相对时间以用户设备当前时间为基准。
- 默认时区为 `Asia/Shanghai`。
- “上午”默认 09:00。
- “下午”默认 15:00。
- “晚上”默认 20:00。
- “晚饭前”默认 18:00。
- 只有日期、没有具体时间或时段时，需要反问具体时间。
- 用户说“6月13日下午3点”时，解析为当年 6 月 13 日 15:00；如果该日期已经过去，需要反问确认年份或新时间。
- 用户说“提前半小时提醒我”时，事件照常写入内部日历，并额外生成相对事件开始时间的提醒。
- 用户只说“有个会议”且没有时间时，不写入日历，先反问。

## 6. 数据模型

### 6.1 CalendarEvent
```ts
type CalendarEvent = {
  id: string;
  title: string;
  startsAt: string;
  endsAt: string | null;
  location: string | null;
  notes: string | null;
  status: "active" | "cancelled";
  sourceText: string;
  createdAt: string;
  updatedAt: string;
};
```

### 6.2 Reminder
```ts
type Reminder = {
  id: string;
  title: string;
  remindAt: string;
  relatedEventId: string | null;
  status: "active" | "done" | "cancelled";
  sourceText: string;
  createdAt: string;
  updatedAt: string;
  notifiedAt: string | null;
};
```

### 6.3 ConversationMessage
```ts
type ConversationMessage = {
  id: string;
  role: "user" | "assistant";
  content: string;
  createdAt: string;
};
```

### 6.4 本地存储策略
- MVP 可使用浏览器本地存储或本地数据库实现。
- 用户日历事件、提醒和对话消息保存在本地。
- 若接入云端 AI 服务，只发送当前用户输入、当前时间、时区和必要澄清上下文。
- 不上传完整日历库和完整提醒库。

## 7. 业务流程

### 7.1 新建日历事件
```text
用户输入
  ↓
AI Agent 解析为结构化 JSON
  ↓
检查是否缺少事件时间
  ↓
完整 → 写入 CalendarEvent → 返回确认
缺失 → 反问时间 → 用户补充 → 重新解析
```

### 7.2 新建提醒
```text
用户输入
  ↓
AI Agent 解析提醒内容和提醒时间
  ↓
完整 → 写入 Reminder → 返回确认
缺失 → 反问时间或内容
```

### 7.3 为事件创建提前提醒
```text
用户输入
  ↓
AI Agent 解析事件时间和提前提醒规则
  ↓
写入 CalendarEvent
  ↓
根据 reminder_offset_minutes 计算 remindAt
  ↓
写入 relatedEventId 指向事件的 Reminder
```

### 7.4 修改记录
```text
用户输入
  ↓
AI Agent 解析修改意图和匹配条件
  ↓
查询 CalendarEvent / Reminder
  ↓
0 个匹配 → 返回未找到
1 个匹配 → 更新记录 → 返回确认
多个匹配 → 列出选项 → 用户选择 → 更新
```

### 7.5 删除记录
```text
用户输入
  ↓
AI Agent 解析删除意图和匹配条件
  ↓
查询 CalendarEvent / Reminder
  ↓
0 个匹配 → 返回未找到
1 个匹配 → 标记为 cancelled → 返回确认
多个匹配 → 列出选项 → 用户选择 → 标记为 cancelled
```

### 7.6 查询日历
```text
用户输入
  ↓
AI Agent 解析时间范围
  ↓
查询 active CalendarEvent 和 active Reminder
  ↓
按时间升序展示
```

### 7.7 提醒触达
```text
定时检查 active Reminder
  ↓
找到 remindAt <= 当前时间 且 notifiedAt 为空的提醒
  ↓
触发浏览器通知
  ↓
写入 notifiedAt
```

## 8. 界面设计

### 8.1 信息架构
- 对话页：默认首页，用户输入和确认结果都在这里完成。
- 日历页：展示 App 内部日历，支持按天查看事件和提醒。
- 提醒页：展示独立提醒和事件关联提醒，支持完成、取消和删除。
- 设置页：通知权限、隐私说明、数据清除。

### 8.2 首次使用
- 首屏直接进入对话页。
- 不在启动时请求通知或麦克风权限。
- 用户第一次创建提醒时，请求通知权限。
- 用户拒绝通知权限后，不反复弹窗，只在设置页提示状态。

## 9. 非功能需求

### 9.1 性能
- 对话响应时间：P95 < 3 秒。
- 本地查询：P95 < 100ms。
- 通知检查延迟：P95 < 60 秒。

### 9.2 安全与隐私
- 本地保存日历事件、提醒和对话消息。
- API 调用使用 HTTPS。
- 调用 AI 服务时，只发送当前输入、当前时间、时区和必要澄清上下文。
- 不上传完整日历库、完整提醒库。
- 设置页提供清除本地数据入口。

### 9.3 可用性
- 用户可以在 10 秒内完成一条日历事件记录。
- 用户可以在 10 秒内完成一条提醒创建。
- 反问只问一个问题。
- 回复简洁，不主动延伸话题。

### 9.4 兼容性
- 支持现代桌面和移动浏览器。
- 支持深色模式。
- Web Notification 不可用时，功能降级为 App 内提醒状态展示。

## 12. 风险与挑战

### 12.1 技术风险
- **AI 解析不稳定**：中文时间表达多样，可能识别错。
  - 缓解：使用结构化 JSON、明确时间规则、补充验收用例。
- **通知能力受浏览器限制**：不同浏览器对通知权限和后台触达支持不同。
  - 缓解：MVP 提供 App 内提醒状态作为降级方案。
- **隐私与 AI API 冲突**：用户输入需要发给 AI 服务解析。
  - 缓解：只发送当前输入和必要上下文，不上传完整日历库。

### 12.2 产品风险
- **被误解为时间规划工具**：如果界面出现任务优先级、番茄钟、效率洞察，产品会偏离核心。
  - 缓解：P0 只围绕“记录、提醒、查询、改删”设计。
- **记录和提醒边界不清**：用户可能不理解“事件”和“提醒”的区别。
  - 缓解：对用户统一表达为“已记录/已提醒”，内部数据模型再区分。

### 12.3 求职展示风险
- **功能做得多但不深**：容易变成浅层 Demo。
  - 缓解：P0 聚焦文字输入、结构化解析、内部日历、提醒、本地存储和可演示闭环。

## 13. 成功指标

### 13.1 求职场景
- 能演示 5 个核心流程：记录日历事件、缺失时间反问、创建提醒、提前提醒、查询/修改/删除记录。
- 至少 3 个内测用户反馈“这个能用”。
- 能在面试中清晰讲出：产品定位、MVP 边界、AI Agent 解析方案、数据模型、隐私策略、迭代计划。

### 13.2 产品场景
- 内测用户次日留存 > 50%。
- 平均每个用户每天记录 3 次以上。
- 30% 以上内测用户愿意推荐给朋友。

## 14. 附录

### 14.1 术语表
- **内部日历**：Timely 自己保存和展示的日历数据，不等同于系统日历。
- **日历事件**：用户要记录在内部日历里的事情，例如会议、体检、聚餐。
- **提醒**：到某个时间点通知用户的事项，可以独立存在，也可以关联某个日历事件。
- **AI Agent**：负责理解用户自然语言并输出结构化 JSON 的解析层。
- **反问**：AI 在信息不完整时主动询问用户。
- **沉默原则**：不主动推送任何用户没问的信息。
