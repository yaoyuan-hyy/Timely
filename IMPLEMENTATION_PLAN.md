# Timely 搭建计划

> 依据：`PRD.md` v0.3
> 创建日期：2026-06-10
> 用途：后续 AI coding 的主计划。代码生成应优先围绕本文档拆分任务，并保持与 `PRD.md` 一致。

---

## 1. 建设目标

Timely 的 MVP 目标不是时间规划工具，而是一个“自然语言记录 + 内部日历 + 提醒”的个人助手。

第一阶段要证明的核心闭环是：

```text
用户输入一句自然语言
  ↓
AI Agent 或本地解析器输出结构化结果
  ↓
程序写入内部日历或提醒
  ↓
用户能在 App 内查看、查询、修改、删除
  ↓
到点时提醒能触达或在 App 内降级展示
```

后续所有代码生成都应服务这个闭环。任何任务优先级、番茄钟、效率洞察、时间规划建议，都不进入 MVP。

---

## 2. 搭建原则

- **先闭环，后智能**：先用本地规则/模拟 Agent 跑通创建、查询、改删、提醒，再接真实 AI API。
- **先内部日历，后系统集成**：MVP 只做 Timely 自己的数据和界面，不接 Apple Calendar / Google Calendar。
- **先单条主要意图，后多意图**：MVP 一次输入只处理一个主要意图，避免过早复杂化。
- **AI 只解析，程序执行**：AI Agent 只能输出结构化 JSON，不能直接写数据。
- **所有写入都可验证**：每次创建、修改、删除后，页面状态和本地数据都应可见。
- **提醒不依赖通知权限才成立**：浏览器通知失败或被拒绝时，提醒仍保存并在 App 内展示状态。

---

## 3. 推荐技术路线

### 3.1 前端形态

- 继续使用现有 Web 项目作为 MVP 载体，但界面按手机客户端优先设计。
- 首页直接进入可用 App，不做营销落地页。
- 主要界面为：
  - 对话页：自然语言输入、澄清、确认结果。
  - 日历页：按天查看内部日历事件。
  - 提醒页：查看独立提醒和事件关联提醒。
  - 设置页：通知权限、隐私说明、清除本地数据。

### 3.2 数据持久化

MVP 优先使用浏览器本地存储。推荐统一封装存储层，避免组件直接读写 `localStorage`。

核心数据：

```ts
type TimelyState = {
  events: CalendarEvent[];
  reminders: Reminder[];
  messages: ConversationMessage[];
  pendingClarification: PendingClarification | null;
};
```

### 3.3 Agent 接入策略

分两步实现：

1. **本地解析器阶段**
   - 用规则或模拟函数处理 PRD 验收用例。
   - 输出与真实 AI Agent 一致的结构化 JSON。
   - 目标是先验证产品流程和 UI。

2. **真实 AI Agent 阶段**
   - 新增服务层封装 AI API。
   - 输入只包含当前用户输入、当前时间、时区和必要澄清上下文。
   - 输出必须经过 JSON schema 校验后才能执行。

## 4. 模块拆分

### 4.1 类型层

建议文件：

- `lib/types.ts`

职责：

- 定义 `CalendarEvent`。
- 定义 `Reminder`。
- 定义 `ConversationMessage`。
- 定义 `AgentIntent`、`AgentParseResult`、`PendingClarification`。
- 定义通知权限、记录状态等枚举类型。

### 4.2 时间工具层

建议文件：

- `lib/time.ts`

职责：

- 当前时间与时区处理。
- 日期范围计算，例如“今天”“明天”“6月13日”。
- 默认时段规则，例如上午 09:00、下午 15:00、晚上 20:00。
- 判断时间是否过期。
- 提醒提前量计算，例如提前半小时。

### 4.3 Agent 解析层

建议文件：

- `lib/agent/schema.ts`
- `lib/agent/local-parser.ts`
- `lib/agent/execute.ts`

职责：

- `schema.ts`：定义 Agent 输出结构和校验规则。
- `local-parser.ts`：先覆盖 MVP 验收用例的本地解析器。
- `execute.ts`：把结构化结果转换成状态变更，但不包含 UI 逻辑。

### 4.4 数据层

建议文件：

- `lib/store.ts`
- `hooks/use-timely-state.ts`

职责：

- 统一读写 Timely 本地状态。
- 提供新增、修改、取消、删除、查询事件和提醒的方法。
- 处理数据版本号和未来迁移。

### 4.5 查询与匹配层

建议文件：

- `lib/calendar-queries.ts`
- `lib/matching.ts`

职责：

- 按时间范围查询事件和提醒。
- 修改/删除时根据标题、时间、关键词匹配候选记录。
- 多条匹配时返回选项，交给对话页反问。

### 4.6 通知层

建议文件：

- `lib/notifications.ts`
- `hooks/use-reminder-ticker.ts`

职责：

- 请求浏览器通知权限。
- 定时检查到期提醒。
- 发送浏览器通知。
- 通知不可用时更新 App 内提醒状态。

### 4.7 UI 组件层

建议文件：

- `components/timely-app.tsx`
- `components/chat-panel.tsx`
- `components/calendar-view.tsx`
- `components/reminder-view.tsx`
- `components/settings-view.tsx`
- `components/app-nav.tsx`

职责：

- 保持首屏就是可操作 App。
- 用标签页或底部导航切换对话、日历、提醒、设置。
- 对话页承担创建、澄清、确认。
- 日历页和提醒页承担可视化验证。

## 5. 分阶段实施计划

### Phase 0：产品方向重置

目标：确保代码和文档不再沿用“时间规划/番茄钟/效率洞察”方向。

任务：

- 检查现有页面、类型、种子数据和文案。
- 删除或替换任务优先级、focus timer、insights 等旧概念。
- 更新 README，使其与 `PRD.md` 和本文档一致。
- 保留可复用的基础设施，例如本地存储 hook、布局样式、图标体系。

验收：

- 首页不再呈现时间规划或番茄钟功能。
- README、PRD、搭建计划三者定位一致。

### Phase 1：数据模型和本地存储

目标：建立 Timely 的数据底座。

任务：

- 在 `lib/types.ts` 定义 PRD 中的数据模型。
- 建立 `TimelyState`。
- 实现本地存储读写。
- 实现事件和提醒的新增、更新、取消、删除、查询方法。
- 准备少量 demo 数据，便于开发 UI。

验收：

- 刷新页面后数据不丢失。
- 可以通过内部函数创建事件和提醒。
- 取消记录使用状态更新，不直接硬删除核心记录。

### Phase 2：App 骨架和核心界面

目标：让用户看到 Timely 的新产品形态。

任务：

- 搭建对话、日历、提醒、设置四个主视图。
- 对话页提供输入框、发送按钮、消息列表。
- 日历页按日期展示事件。
- 提醒页展示 active / done / cancelled 状态。
- 设置页展示通知权限、隐私说明、清除数据入口。

验收：

- 首屏可以直接输入自然语言。
- 日历页和提醒页能看到本地状态。
- 移动端和桌面端基本可用。

### Phase 3：本地 Agent 解析器

目标：不依赖外部 AI，先跑通 PRD 的主要验收用例。

任务：

- 实现 `AgentParseResult` 结构。
- 实现本地解析器，优先覆盖：
  - “帮我记录一下6月13号下午3点有一个会议”
  - “6月13日下午3点有个会议”
  - “帮我记录一个会议”
  - “明天下午3点提醒我取快递”
  - “6月13日下午3点有个会议，提前半小时提醒我”
  - “6月13号有什么安排”
  - “最近怎么样”
- 对缺少时间的输入生成 `needs_clarification`。
- 对过期时间生成确认或反问，不直接写入。

验收：

- PRD 验收用例 1-6 和 11 可以通过 UI 手动验证。
- Agent 输出和 PRD 的 JSON 结构一致。

### Phase 4：创建事件和提醒闭环

目标：让一句话真正写入内部日历或提醒。

任务：

- 对话页发送输入后调用本地 Agent。
- `create_event` 写入 `CalendarEvent`。
- `create_reminder` 写入 `Reminder`。
- 带 `reminder_offset_minutes` 的事件同时创建关联提醒。
- 成功后写入 assistant 确认消息。

验收：

- 输入“帮我记录一下6月13号下午3点有一个会议”后，日历页出现会议。
- 输入“明天下午3点提醒我取快递”后，提醒页出现取快递。
- 输入“6月13日下午3点有个会议，提前半小时提醒我”后，日历页有事件，提醒页有 14:30 提醒。

### Phase 5：澄清对话

目标：支持一次任务内的短澄清上下文。

任务：

- 增加 `pendingClarification`。
- 用户输入信息不完整时，不写入数据，只保存待澄清上下文。
- 用户下一轮回答时，合并上轮上下文重新解析。
- 澄清只问一个问题。

验收：

- 用户输入“帮我记录一个会议”时，系统回复“什么时候？”。
- 用户继续输入“6月13日下午3点”后，创建会议事件。
- 澄清完成后清空 `pendingClarification`。

### Phase 6：查询、修改、取消

目标：完成记录管理闭环。

任务：

- 实现 `query_calendar`。
- 实现 `update_event`。
- 实现 `delete_event` 或取消事件。
- 实现候选匹配。
- 多条匹配时列出选项并等待用户选择。

验收：

- “6月13号有什么安排”能返回当天事件。
- “6月13日下午3点的会议改到4点”能更新事件时间。
- “取消6月13日下午3点的会议”能取消单条匹配事件。
- 多条会议匹配时不直接取消，先反问。

### Phase 7：提醒触达

目标：提醒不只是记录，也能到点触达。

任务：

- 实现提醒检查定时器。
- 到点提醒触发浏览器通知。
- 通知权限未授权时，在 App 内标记到期提醒。
- 设置页支持查看通知权限状态。

验收：

- 即将到期的提醒能被检查到。
- 浏览器允许通知时能弹出通知。
- 浏览器拒绝通知时，提醒仍保留并在 App 内可见。

### Phase 8：真实 AI Agent 接入

目标：把本地解析器替换或升级为真实 AI Agent，但不破坏已跑通的执行层。

任务：

- 设计 AI prompt，强调只输出 JSON。
- 增加 API 调用层。
- 增加 JSON schema 校验。
- 校验失败时返回可恢复错误，不写入数据。
- 保留本地解析器作为开发 fallback 或测试 fixture。

验收：

- 真实 AI 输出仍通过同一个执行层写入数据。
- AI 解析失败不会写坏本地状态。
- 发送给 AI 的上下文符合 PRD 隐私要求。

### Phase 9：QA、演示和内测准备

目标：让产品可演示、可解释、可迭代。

任务：

- 把 PRD 的 MVP 验收用例转为手动测试清单。
- 修复移动端布局问题。
- 准备 5 个核心演示流程：
  - 记录日历事件。
  - 缺失时间反问。
  - 创建独立提醒。
  - 创建事件提前提醒。
  - 查询、修改、取消记录。
- 更新 README 的运行说明和产品说明。

验收：

- 新用户打开页面后能在 10 秒内完成一条记录。
- 5 个核心演示流程稳定可走通。
- README 能清楚说明产品定位和运行方式。

## 6. 推荐 AI Coding 任务顺序

后续生成代码时，建议按以下顺序一项一项推进：

1. 重置旧产品 UI 和 README 文案。
2. 定义 `CalendarEvent`、`Reminder`、`ConversationMessage`、`TimelyState`。
3. 实现本地存储和基础 CRUD。
4. 搭建对话、日历、提醒、设置四个视图。
5. 实现本地 Agent 解析器和结构化结果。
6. 打通创建日历事件。
7. 打通创建独立提醒。
8. 打通事件关联提前提醒。
9. 打通澄清对话。
10. 打通查询日历。
11. 打通修改和取消事件。
12. 打通提醒触达和通知权限。
13. 接入真实 AI Agent。
14. 做端到端验收和 README 更新。

每次 coding 最好只选择一个小闭环，例如“输入一句话创建事件并在日历页显示”，不要同时改 UI、Agent、通知和真实 API。

---

## 7. 每次代码生成前的检查清单

在开始新一轮 AI coding 前，先确认：

- 当前任务属于哪个 Phase。
- 是否会引入 PRD 明确不做的功能。
- 是否需要改数据模型。
- 是否需要迁移旧本地数据。
- 是否能用页面或测试验证结果。
- 是否会影响已有创建、查询、提醒闭环。

---

## 8. MVP 完成定义

当以下条件都满足时，可以认为 Timely MVP 初版完成：

- 用户可以通过自然语言创建日历事件。
- 用户可以通过自然语言创建独立提醒。
- 用户可以为事件创建提前提醒。
- 用户可以查看内部日历。
- 用户可以查看提醒列表。
- 用户可以查询某天安排。
- 用户可以修改和取消单条匹配事件。
- 多条匹配时系统会反问，不误删误改。
- 缺少关键时间时系统会反问，不写入脏数据。
- 到点提醒能浏览器通知，或在通知不可用时 App 内降级展示。
- 本地刷新后数据仍保留。
- 产品界面和文案不再表达时间规划、番茄钟、任务优先级或效率洞察。

---

## 9. 暂不进入 MVP 的内容

以下功能只在 P1/P2 之后再讨论：

- 语音输入。
- 重复事件和重复提醒。
- 系统日历同步。
- 多端同步。
- Apple Watch 或桌面小组件。
- 轻量记账。
- 统计回顾。
- 多人共享日历。
- 任何主动时间规划建议。
